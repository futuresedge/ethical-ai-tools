# Nexus Design Review: Open Questions and Unresolved Issues

**The Policy Engine remains abstract.** We defined what it needs to do — evaluate `state_transitions` rows on lifecycle events, fire `spawned_action`, send `tools/list_changed` — but not how it runs in a SQLite + STDIO environment. SQLite has no native event system. The implementation options (polling loop, file watcher, VS Code task trigger) weren't examined. This is the next most important technical question after the experiment phases are complete. 

**The proof template authorship question was raised but not resolved.** Who writes the proof template — Task Owner (upstream, during task spec creation) or Task Performer (first action before implementation)? Both were discussed at different points in the session without settling on one. The experiment will surface the right answer empirically, but the design ambiguity should be named before the experiment runs so the test is deliberate, not accidental. 

**The observable stream format is still undefined.** Plain English events were described and examples given, but the concrete format — how events are stored, how Pete reads them (CLI query? file tail? VS Code panel?), what the stream looks like for a completed task — was never specified. Phase 5 of the experiment tests stream legibility, but without a defined format, the test has no baseline.

**The webhook error handling was skimmed.** The experiment steps include registering a webhook and processing payloads — but what happens if the webhook endpoint is down, or if ngrok drops, or if a GitHub event fires during local server restart? Retry logic, idempotency (GitHub can deliver the same event multiple times), and the `X-GitHub-Delivery` deduplication header were not addressed. These are not Phase 0 blockers but are easy to get wrong early if not named. [lobehub](https://lobehub.com/mcp/ajaychinthapalli-github-webhooks-mcp-server)

**The context card generation process is still hand-wavy.** Context cards are described as pre-generated by the system before agent instantiation, pulling from the knowledge base. But in the experiment, the knowledge base starts empty. The first few context cards will be written manually by Pete. The transition from manual cards to generated cards — what triggers it, what the knowledge base needs to contain before generation is useful — was not designed.


## Open Questions Carried Forward

| Question | Urgency | Where it surfaces |
|---|---|---|
| How does the Policy Engine run without a persistent process in STDIO? | High | After Phase 2 of experiment |
| Who authors the proof template — Task Owner or Task Performer? | Medium | Phase 5 experiment pass/fail |
| What is the concrete format and display mechanism for the observable stream? | Medium | Phase 5 experiment design |
| How does context card generation bootstrap from an empty knowledge base? | Medium | First real sprint with Nexus live |
| Does VS Code handle 10+ dynamically generated agent specs without degradation? | Low | Phase 3 experiment at scale |
| What is the webhook retry/idempotency strategy for Nexus? | Low | Before production use |
