# Nexus Experiment 1 — Agent & Skills Plan

**Date:** 2026-03-01  
**Status:** DRAFT  
**Scope:** Agents required to build and operate the Nexus experiment (Exp-01, Phases 1–5)

> Current framework operating protocol continues unchanged while the experiment runs.  
> This plan runs in parallel — it does not replace or modify any existing pipeline agent.

---

## Two Categories of Agent

| Category | When needed | Purpose |
|---|---|---|
| **Build Agents** | Now — during the experiment | Implement the Nexus MCP server and infrastructure across Phases 1–5 |
| **Nexus Operational Agents** | Emerge from Phase 3 onwards | The agent roles Nexus orchestrates once running |

---

## Part 1 — Build Agents (who does the work in the experiment)

These agents are invoked using the current VS Code agent protocol.  
They exist to BUILD Nexus — they are not Nexus agents themselves.

---

### Agent 1.1 — Nexus Server Builder

**Phases:** 1 (Foundation) · 2 (OCAP Tools) · 3 (Agent Spec Generation)

**Creates:**
- `nexus/server.ts` — MCP server (STDIO transport)
- `nexus/schema.sql` — SQLite schema: tasks, documents, audit_log, stream_events, tool_registry
- `nexus/db.ts` — SQLite wrapper (better-sqlite3)
- `.vscode/mcp.json` — VS Code server registration
- `.github/agents/task-performer.template.md` — per-task spec template (Phase 3)
- `nexus/` scaffolding via `package.json`, `tsconfig.json`

**Skills needed:**

| Skill | Why |
|---|---|
| TypeScript strict mode | All Nexus code is TypeScript; Zod schemas enforce tool parameters |
| `@modelcontextprotocol/sdk` | `McpServer`, `server.tool()`, STDIO transport, `notifications/tools/list_changed` |
| `better-sqlite3` | Synchronous SQLite; prepared statements; WAL mode |
| Zod | Tool parameter validation — `reason` must be required, never optional |
| OCAP tool naming grammar | `{verb}_{subject}_{task_slug}` — must follow `NexusToolGrammar.md` exactly |
| Dynamic dispatch pattern | Registering task-scoped tools at runtime, not only at startup |
| Agent spec template format | Generating valid `.agent.md` files from a `{{PLACEHOLDER}}` template |
| **`nexus-ontology`** | Before designing any tool or schema table, check all 6 ONE dimensions — see Part 5 |

**Does NOT need:** HTTP, Fastify, GitHub API, ngrok (those are Agent 1.2)

## Part 2 — Nexus Operational Agents (the roles Nexus orchestrates)

These agents are the OUTPUT of the experiment, not the builders of it.  
They are instantiated by Nexus, not the current framework pipeline.

---

### Agent 2.1 — Task Performer (dynamically generated)

**Generated by:** `activate_task` tool (Phase 3)  
**File pattern:** `.github/agents/task-performer-{task-id}.agent.md`  
**Lifecycle:** Created on activation · Deleted on `deactivate_task`

**Tool set at instantiation:**

| Tool | Verb | Subject | Side effects |
|---|---|---|---|
| `read_task_spec_{slug}` | read | task_spec | None |
| `write_proof_template_{slug}` | write | proof_template | Audit log entry |
| `append_work_log_{slug}` | append | work_log | Audit log entry |
| `submit_proof_{slug}` | submit | proof | Write + state transition + stream event (compound) |
| `get_context_card` | get | context_card | None (universal) |
| `raise_uncertainty` | raise | uncertainty | Stream event (universal) |

**Behavioural constraints (baked into the template, not a loadable skill):**
- Begin every task: call `get_context_card` before any other tool
- If context card is insufficient: call `raise_uncertainty` — never guess
- Never reference a tool outside its own `{task_slug}`
- Proof template must be written before any implementation begins

**Skills loaded:** None at experiment stage — context card carries all task-specific knowledge.  
Post-adoption: load `nexus-tool-grammar` for agents that may add tools.

## Part 5 — The ONE Ontology as Build Constraint

### Why a skill, not an agent

The ONE Ontology's 6 dimensions (Capability, Accountability, Quality, Temporality, Context, Artifact) are a **design rubric**, not a runtime role. They are applied at the moment a tool, a schema table, or an agent spec is being designed — not during task execution. An "Ontology Reviewer" agent would be appropriate post-adoption (analogous to the Agent Spec Reviewer in the current pipeline), but during the experiment it creates coordination overhead without payoff. The rubric belongs in a skill that the Nexus Server Builder loads and applies inline.

### Coverage map — Phase 1–5 vs. the 6 dimensions

| Dimension | Phase 1–5 coverage | Gap type |
|---|---|---|
| **Capability** | ✅ Full — OCAP tool registration is Phase 2's core deliverable | None |
| **Accountability** | ✅ Full — `audit_log` in schema; every compound tool writes an entry; GitHub webhooks write with `actor: 'webhook:github'` | None |
| **Quality** | ⚠️ Skeleton — `submit_proof` validates state preconditions; proof template is written before implementation (structural). QA rules table and structured rule evaluation are deferred | Intentional deferral — QA Reviewer (Agent 2.2) is a future phase |
| **Temporality** | ⚠️ Hardcoded — state transitions are inside tool implementations, not a `state_transitions` table evaluated by a Policy Engine | Intentional simplification — 3 states and 2 transitions are manageable inline; must be flagged as technical debt, not final architecture |
| **Context** | ⚠️ Thin — `get_context_card` returns raw task row, not a generated briefing from a knowledge base | Intentional deferral — Assumption B3, tested in a later probe |
| **Artifact** | ⚠️ Skeleton — `documents` table has `doc_type` + `version`, but no lineage graph (no `document_lineage` table) | Intentional deferral — proof-of-concept only; lineage traversal is post-adoption |

**Pass criterion for the experiment is NOT full ontology coverage.** Capability and Accountability must be complete. Quality, Temporality, Context, and Artifact are covered enough to prove the structural principle — not enough to operate at production quality. The gaps are named explicitly here so build agents don't gold-plate the experiment, and so Pete can evaluate the adoption decision knowing exactly what is and isn't proven.

### 6-dimension design checklist (for the Nexus Server Builder)

Apply this before implementing any tool, any schema table, or any agent spec section:

| Dimension | Question to answer | If the answer is "no" or "unclear" |
|---|---|---|
| **Capability** | Does tool possession alone enforce the boundary — with no runtime identity check? | Redesign: the access model is wrong |
| **Accountability** | Does every mutation write an `audit_log` entry inside the tool, not as an instruction to the agent? | Move the audit write inside the tool implementation |
| **Quality** | Is the acceptance condition for this tool's output encoded in the tool (state precondition, Zod schema, content hash) rather than in agent instructions? | Harden the tool — don't rely on agent compliance |
| **Temporality** | Does the tool check the task's current state before executing and reject with a typed error if the precondition is not met? | Add the precondition check — no tool should be lifecycle-agnostic |
| **Context** | Will the agent calling this tool have been given what it needs (via context card) before it makes this call? | Ensure `get_context_card` precedes this tool in the task flow |
| **Artifact** | Does the document this tool creates or reads have a declared type, a write mode, and a version entry in the `documents` table? | The subject vocabulary entry is incomplete — fix before building the tool |

### What the ontology is NOT

- It is not a runtime check agent — no agent reviews tool calls against the 6 dimensions during execution
- It is not a gate in the current pipeline — framework pipeline agents (Zone 1–5) continue operating under existing rules
- It is not a replacement for the Tool Grammar — the grammar governs naming; the ontology governs structural completeness; both apply together

### Post-adoption: when an Ontology Reviewer agent becomes warranted

If the experiment succeeds and Nexus is adopted, the ontology should be enforced by a dedicated **Nexus Tool Reviewer** agent — analogous to the existing Agent Spec Reviewer. It would read proposed tool specs and output a structured verdict against the 6-dimension checklist before any tool is implemented. That agent is NOT part of the experiment.
