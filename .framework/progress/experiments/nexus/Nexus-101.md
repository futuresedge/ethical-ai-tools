## What Is Nexus?

Software development with AI agents introduces a coordination problem that most current tooling ignores. When multiple agents are working simultaneously — one defining requirements, one writing code, one verifying quality — there is no structural guarantee that they stay within their lane. An agent instructed to "only work on task-07" can still, through a poorly formed tool call or an ambiguous context, write to the wrong file, overwrite another agent's work, or report success on a test that ran in the wrong environment. Most frameworks respond to this with longer instructions: more `NEVER` clauses in agent specs, more rules in system prompts. Nexus takes a different position entirely. Instead of telling agents what not to do, it makes the violations structurally impossible. 

The mechanism is borrowed from a principle in computer security called the Object Capability Model. The idea is simple: a capability is not a permission you check against a list — it is a reference you either possess or don't. If an agent doesn't have a reference to a tool, it cannot call that tool, and there is no check to bypass, no rule to violate, no instruction to ignore. The violation is architecturally absent. In Nexus, every tool is named after exactly what it does and who it belongs to. A Task Performer assigned to task-07 receives a tool called `write_proof_template_task_07` — and only that task's tools. The tool for task-08 does not appear in its registry. VS Code's agent mode enforces this at the client layer before any call reaches the server. No agent can exceed its scope because there is nothing to call outside it. 

## Typed Artefacts and Structural Governance

The second load-bearing idea is that every document an agent produces has a declared type — `task_spec`, `proof`, `qa_review`, `work_log`, `acceptance_criteria` — and that type determines which agents can read it, which can write it, and under what lifecycle conditions. A proof document, for example, is write-once: it can only be submitted (not edited) through a compound tool called `submit_proof`, which atomically writes the document, transitions the task's state from `CLAIMED` to `PROOFSUBMITTED`, and emits a plain-English event to a stream Pete can watch in real time. The agent cannot write the proof and forget to update the state, because those two operations are the same tool call. Mandatory side effects live inside the tools, not in instructions to the agents. 

This typed artefact system creates a natural grammar for the entire tool catalogue. Every Nexus tool is named `{verb}_{subject}_{task_scope}` — where verbs are a closed set of eight operations (read, write, append, submit, request, search, get, raise) and subjects are the document type taxonomy. Adding a new artefact type means adding a row to a matrix; adding a new agent role means assigning a subset of existing tools. The catalogue stays organised as it grows because the grammar, not convention, keeps it coherent. 

## A Single Source of Truth Across Two Systems

The third problem Nexus solves is observability across a system where not everything flows through a single chokepoint. Agents working on code use the GitHub MCP server to create branches, push commits, and open pull requests — operations that Nexus doesn't execute and shouldn't intercept. But Nexus needs to know they happened. The solution is GitHub's native webhook mechanism: every push, pull request, review, and merge fires a payload to a Nexus endpoint. The branch naming convention — `task/task-07` — gives Nexus the join key it needs to attribute every git event to the right task. The result is a single audit log covering both framework artefacts written through Nexus tools and code artefacts managed through GitHub, all queryable by `task_id`, all with a clear record of who did what and when. [lobehub](https://lobehub.com/mcp/ajaychinthapalli-github-webhooks-mcp-server)

Pete, as the human Product Owner, experiences this as a plain-English stream of events: agent instantiated, proof template written, branch created, tests passed, QA review submitted, PR merged, task complete. He can reconstruct the full chain of custody for any task without opening a single file. He approves gates; he doesn't manage the coordination. The framework's entire purpose is to make that possible — to make AI agents trustworthy not through better instructions, but through a structure that makes trustworthy behaviour the only available behaviour.
